# Chepherd v0.9.1 — Component Inventory + Agent-to-Agent Sequence

**What this is.** Canonical vocabulary for chepherd's v0.9.1 target architecture. Every process, state store, kernel object, network endpoint, connection, wire format, and human role named once with its category, role, location, and relationships. Use these names + numbers for all future architecture discussions.

**Authority.** Authored 2026-05-28 from architect deliberation in [#205](https://github.com/chepherd/chepherd/issues/205) (with parent context in [#186](https://github.com/chepherd/chepherd/issues/186)). Reflects v0.9.1 target shape — legacy items (regex `@-relay`, nested-podman, daemon-holds-PTY-master shortcut, separate internal/external messaging paths) explicitly retired.

## What's RETIRED in v0.9.1 (gone, not in the table below)

| Retired thing | Replaced by |
|---|---|
| Regex `@-relay` (`internal/messagebus/relay.go`) | A2A `tasks/send` via runner's A2A server |
| Daemon-holds-PTY-master-FDs shortcut | Runner-owns-PTY-master per pod |
| Nested-podman invention (`--root /var/lib/chepherd-agents/...`) | Standard pod-per-agent (K8s API or sibling podman) |
| `chepherd.send_to_session` as the user-facing tool name | `chepherd.send_to_peer` (unified) |

| Optional `chepherd-bridge` (stdio-MCP adapter) | Direct HTTP-MCP from agent to runner's local Unix socket (all target agents support HTTP-MCP) |

## What's A2A standard vs RFC NAT-traversal vs chepherd-specific

The patterns below mix three categories of mechanism. Knowing which is which is critical — we follow standards where they exist, and invent only where they don't:

| Layer | What | Source |
|---|---|---|
| Agent Card at `/.well-known/agent.json` | Per-agent discovery metadata (capabilities, endpoint URL, auth schemes, public key) | **A2A standard** — we follow |
| Agent Card directory (broker hosts a catalog) | Aggregated discovery across many agents | **chepherd deployment choice** — A2A-compatible extension (directory just stores standard cards) |
| `tasks/send`, `tasks/get`, task lifecycle | The application-layer message envelope | **A2A standard** — we follow |
| JWT Bearer / mTLS / OAuth in headers | Auth on A2A calls | **A2A standard** — A2A spec defines these auth schemes; we use JWT Bearer for cross-org |
| ICE candidate gathering (host, srflx, relay) | Discover all possible network paths between two peers | **RFC 8445 (ICE)** — same stack browsers + WebRTC use |
| STUN binding | Discover own public IP/port behind NAT | **RFC 5389 (STUN)** |
| TURN allocate | Get a relay-assigned transport address as fallback | **RFC 5766/8656 (TURN)** |
| SDP offer/answer over signaling | Exchange ICE candidates, DTLS fingerprints, ICE credentials | **RFC standard** (WebRTC, JSEP) |
| DTLS handshake | E2E encrypt the data channel | **RFC 6347/9147 (DTLS)** |
| WebRTC DataChannel | P2P framed messaging on top of DTLS | **RFC 8831** |
| Knock marker (`<<CHEPHERD-MSG task=... from=...>>`) + `chepherd.get_task` fetch | Inject A2A task into a CLI agent's PTY without breaking the TUI | **chepherd-specific** — needed because coding agents read prompts from stdin, not native A2A |
| Per-org audit log | Local audit storage per organization | **chepherd-specific** |

**Rule:** if a job already has an RFC or an A2A field, use the standard. Don't invent chepherd-specific machinery for it. The only purely-chepherd inventions are the knock-marker (because coding agents need stdin injection) and the per-org operational layer (audit, RBAC, federation glue).

## The three federation patterns — overview

| # | Pattern | Orgs | Publicly-addressable component | When to use |
|---|---|---|---|---|
| 1 | **Intra-org** | 1 | None | Both agents in the same private org |
| 2 | **Cross-org via 3rd-party public broker** | 3 (Org-X + Org-Y + 3rd Org e.g. OpenOva) | 3rd Org's daemon-S + STUN + TURN + signaling | Both private orgs trust a neutral 3rd party for federation |
| 3 | **Cross-org point-to-point** | 2 (Org-X + Org-Y, one side publicly hosts) | Org-Y's own daemon-Y | One org chose to expose its own daemon; no 3rd party in path |

In **all patterns**: daemons, runners, operators are outbound-only. Publicly-addressable component(s) are the only inbound-reachable endpoints. Runners NEVER directly exposed to internet — even in Pattern 3 where daemon-Y is public, runners stay private and daemon-Y proxies.

## How an agent becomes aware of its peers

**Awareness is operator-driven, seeded at spawn time.** Discovery (finding endpoints + capabilities at runtime) is A2A standard. Two different things:

| Concept | What | When | How |
|---|---|---|---|
| **Awareness** | "agent-B exists and you may talk to it" | At spawn time | Operator-X grants the peering via #7 RBAC; daemon-X writes peer roster into #11 system prompt before fork; agent-A reads this on startup |
| **Discovery** | "agent-C is reachable at this endpoint with these capabilities" | At runtime (per call) | A2A Agent Card fetched from `/.well-known/agent.json` (direct) or from chepherd-broker's directory (aggregated) |

Awareness is the policy gate ("am I allowed to talk to B?"); discovery is the wire mechanism ("where is B and what does it do?"). The patterns below show both explicitly.

### What's NOT used in Pattern 1
- chepherd-relay (#2) — no NAT to traverse
- STUN service (#41), STUN (#42), TURN (#43), ICE (#44) — no public addressing needed
- DTLS (#45) — cluster network already TLS-protected
- WebRTC DataChannel (#33), connection #25 (TURN fallback) — direct HTTPS suffices
- Federation connection (#27) — same org

### Pattern 2 key properties
- **9 phases explicit** — discovery, auth, candidate gathering, SDP, connectivity checks, DTLS, A2A send, knock+fetch, audit
- **Standards used**: A2A for application + Agent Cards + auth; RFC 8445/5389/5766/6347/8831 for transport
- **chepherd-specific only**: directory aggregation, knock marker, per-org audit
- **Broker never sees agent payload** — DTLS E2E even in TURN-fallback (keys never crossed broker)
- **Each org keeps its own audit log** — broker has metadata-only for SLA/billing

```mermaid
    Note over A,C: Phase 0 — Agent Card publishing (A2A standard well-known URL)
    Note right of CPY: operator-Y spawned agent-C
    CPY->>CPY: publish Agent Card for agent-C at CPY public URL slash well-known slash agent.json
    Note right of CPY: A2A standard direct publish, no broker needed
    Note right of CPX: operator-X configured Org-Y trust out of band (URL plus mTLS cert)

    Note over A,C: Phase 1 — Discovery via direct Agent Card fetch (A2A standard)
    A->>RA: discover_peer capability X
    RA->>CPX: query for capability X
    CPX->>CPY: HTTP GET CPY public URL slash well-known slash agents.json
    Note right of CPX: direct fetch over connection 27, mTLS authenticated
    CPY->>CPX: list of agents in Org-Y matching capability X
    CPX->>RA: list including agent-C
    RA->>A: Agent Cards

    Note over A,C: Phase 2 — Credential request direct to home daemon
    A->>RA: send_to_peer agent-C body
    RA->>CPX: request JWT
    CPX->>CPY: federation auth request for agent-C (direct, no broker)
    Note right of CPX: connection 27 outbound HTTPS plus mTLS direct to CPY public endpoint
    CPY->>CPY: check RBAC 7, mint JWT 10 scoped A to C with 60s TTL
    CPY->>CPX: signed JWT
    CPX->>RA: bundled credentials

    Note over A,C: Phases 3 to 6 — SIMPLIFIED (no peer-to-peer needed)
    Note over RA,CPY: runner-A can reach CPY directly (CPY is publicly addressable)
    Note over RA,CPY: no STUN binding, no TURN allocate, no ICE checks
    Note over RA,CPY: standard TLS handshake (one side has a public certificate)

    Note over A,C: Phase 7 — A2A tasks_send direct to CPY public endpoint
    RA->>CPY: A2A tasks_send body with JWT in header
    Note right of RA: outbound HTTPS to CPY public endpoint, standard TLS encryption
    CPY->>CPY: A2A endpoint authenticates JWT signed by CPY own key

    Note over A,C: Phase 7b — CPY proxies to private runner-C (gateway role)
    CPY->>RC: forward A2A call over Org-Y internal cluster network
    Note right of CPY: CPY is the public gateway for its private runners
    RC->>RC: persist task, generate knock marker

    Note over A,C: Phase 8 — Knock plus structured fetch (chepherd-specific, same as Pattern 2)
    RC->>C: write knock to PTY master 13 via kernel 29
    Note over C: agent-C reads knock from stdin
    C->>RC: get_task
    RC->>C: structured A2A task envelope
    Note over C: agent-C processes the message
    RC->>CPY: response
    CPY->>RA: response to A2A tasks_send

    Note over A,C: Phase 9 — Audit (local to each org)
    par audit local
        RA->>CPX: audit sent
    and
        RC->>CPY: audit received
    end
    Note over CPX,CPY: each org has its own audit, no 3rd party
```

### Pattern 3 key properties
- **No 3rd party** — Org-X reaches Org-Y's public endpoint directly
- **Phases 3 to 6 SKIPPED** — daemon-Y is publicly addressable, so standard TLS suffices (no NAT to traverse for the daemon-to-daemon hop)
- **daemon-Y acts as gateway proxy** for its private runner-C — runner-A never directly contacts runner-C
- **Symmetric variant**: if BOTH orgs publicly host, daemon-to-daemon and (optionally) runner-to-runner can both be direct; this is what OpenOva does on its end
- **Trust burden is bilateral**: Org-X and Org-Y must pre-exchange certificates / public keys out of band

---

## Invariants across ALL three patterns

- ❌ No regex parsing of agent stdout — agents express intent via MCP tools only
- ❌ No PTY FDs cross pod boundaries — each runner owns its local PTY
- ❌ Daemons hold NO PTY FDs (only runners do)
- ❌ Runners are NEVER directly internet-exposed — they only initiate outbound; daemon-Y or 3rd Org broker is the public proxy
- ❌ No agent payload EVER in plaintext over the internet — DTLS E2E (Pattern 2) or TLS to a trusted endpoint (Pattern 3)
- ✅ ONE MCP/A2A path for all inter-agent messaging — intra-org and cross-org use the same envelope and tool
- ✅ Each org keeps its own audit log locally (#8); cross-org events have metadata-only at any broker
- ✅ Operator-per-org and browser-per-operator — NO single human can drive across orgs
- ✅ Agent Cards (A2A standard) are the discovery medium in all three patterns
- ✅ Knock-then-fetch is the only chepherd-specific delivery mechanism — every other layer follows A2A or RFC standards
