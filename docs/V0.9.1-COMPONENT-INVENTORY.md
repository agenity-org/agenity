# Chepherd v0.9.1 — Component Inventory + Agent-to-Agent Sequence

**What this is.** Canonical vocabulary for chepherd's v0.9.1 target architecture. Every process, state store, kernel object, network endpoint, connection, wire format, and human role named once with its category, role, location, and relationships. Use these names + numbers for all future architecture discussions.

**Authority.** Authored 2026-05-28 from architect deliberation in [#205](https://github.com/chepherd/chepherd/issues/205) (with parent context in [#186](https://github.com/chepherd/chepherd/issues/186)). Reflects v0.9.1 target shape — legacy items (regex `@-relay`, nested-podman, daemon-holds-PTY-master shortcut, separate internal/external messaging paths) explicitly retired.

---

## What's RETIRED in v0.9.1 (gone, not in the table below)

| Retired thing | Replaced by |
|---|---|
| Regex `@-relay` (`internal/messagebus/relay.go`) | A2A `tasks/send` via runner's A2A server |
| Daemon-holds-PTY-master-FDs shortcut | Runner-owns-PTY-master per pod |
| Nested-podman invention (`--root /var/lib/chepherd-agents/...`) | Standard pod-per-agent (K8s API or sibling podman) |
| `chepherd.send_to_session` as the user-facing tool name | `chepherd.send_to_peer` (unified) |
| Separate MCP path for internal peers + A2A path for external | One A2A path for all inter-agent traffic |
| Optional `chepherd-bridge` (stdio-MCP adapter) | Direct HTTP-MCP from agent to runner's local Unix socket (all target agents support HTTP-MCP) |

## How agent-A becomes aware of agent-B (the awareness flow)

Agents don't discover peers spontaneously. **Awareness is seeded by the chepherd-daemon at spawn time.** This is the prerequisite to any inter-agent traffic.

```mermaid
sequenceDiagram
    autonumber
    participant OP as Operator
    participant BR as Browser
    participant CPX as chepherd-daemon-X
    participant CPY as chepherd-daemon-Y
    participant RA as runner-A
    participant RC as runner-C
    participant A as agent-A
    participant C as agent-C
    participant STUN as STUN-service

    Note over OP,C: Setup spans TWO organizations - org-X and org-Y
    OP->>BR: spawn agent-A in org-X and agent-C in org-Y
    BR->>CPX: spawn agent-A
    BR->>CPY: spawn agent-C

    CPX->>CPY: federation handshake exchange Agent Cards 9
    CPY->>CPX: ack with reciprocal Agent Card

    CPX->>CPX: write session prompt 11 for A with peer agent-C info
    CPY->>CPY: write session prompt 11 for C with peer agent-A info

    par Spawn pod in org-X
        CPX->>RA: create pod with prompt 11
        RA->>CPX: register session sid-A; open control WS 22
        RA->>STUN: STUN binding 42
        STUN->>RA: srflx address (public IP behind NAT)
        RA->>A: fork agent with PTY 12; attach slave 14
        Note over A: agent-A reads prompt 11
        Note over A: now KNOWS agent-C exists in org-Y
    and Spawn pod in org-Y
        CPY->>RC: create pod with prompt 11
        RC->>CPY: register session sid-C; open control WS 22
        RC->>STUN: STUN binding 42
        STUN->>RC: srflx address
        RC->>C: fork agent with PTY 12; attach slave 14
        Note over C: agent-C reads prompt 11
        Note over C: now KNOWS agent-A exists in org-X
    end

    Note over A,C: Both agents aware of each other via seeded prompt 11; ready for inter-agent messaging
```

**The key fact:** "awareness" is not magic — it's a field in #11 system prompt that #1 chepherd-daemon writes at spawn time, derived from #6 session registry + #7 RBAC grants. The agent reads it on startup and treats peer names as legitimate targets it can call.

Three discovery mechanisms (any combination):
- **Seeded at spawn** (default, shown above) — system prompt has the team roster
- **Live query** — agent calls `chepherd.list_peers()` MCP tool at runtime; returns the current roster from #6 session registry
- **Operator instruction** — human types "ask agent-B to review this" into agent-A's pane

## Sequence — agent-A in org-X messages agent-C in org-Y (cross-org federation)

This is the most general case: two agents in different organizations, both behind firewalls, both pods only able to initiate outbound traffic. Includes all NAT-traversal layers (STUN, ICE, TURN, DTLS) and federation between two independent chepherd-daemon deployments.

### Preconditions

- Both agents already spawned in their respective orgs (see Awareness Flow above)
- runner-A in pod-A in org-X cluster; runner-C in pod-C in org-Y cluster
- Each runner holds:
  - Persistent control WS 22 to its OWN chepherd-daemon (runner-A to chepherd-X, runner-C to chepherd-Y)
  - Persistent relay WS 23 to chepherd-relay (public)
  - System prompt 11 with peer roster including the cross-org peer
  - Agent Card 9 published at endpoint 16
- chepherd-X and chepherd-Y have a federation grant for the team (operator-approved per pair)

### Step explainer (numbers match diagrams above)

1. **Initiate** — agent-A reads #11 system prompt (seeded at spawn with team roster naming agent-C in org-Y), calls `chepherd.send_to_peer("agent-C", body)` via #21 (unix socket #15, MCP #30 over AF_UNIX #38).

2. **Discovery + federated auth** — runner-A queries its LOCAL daemon (chepherd-daemon-X) via #22 control WS. Chepherd-daemon-X sees agent-C is in org-Y → makes federation A2A call via #27 (mTLS) to chepherd-daemon-Y. Chepherd-daemon-Y checks ITS #7 RBAC, mints a #10 JWT scoped to this A→C call, returns agent-C's endpoint candidates + JWT. Chepherd-daemon-X bundles + returns to runner-A.

3. **ICE candidate gathering (#44)** — runner-A queries #41 STUN service with #42 binding request; learns its srflx (public-internet-visible) address. Now has candidate set: host (pod NIC), srflx (public via STUN), relay (via #20 + #43 TURN).

4. **Signaling exchange** — runner-A sends SDP offer + all ICE candidates via #23 to #19 (relay signaling). Relay forwards to runner-C via runner-C's own #23. Runner-C gathers its own candidates from #41, replies with SDP answer + candidates; relay shuttles back to runner-A.

5. **Connectivity checks + DTLS** — both sides run #44 ICE connectivity checks (ping every candidate pair via #42 STUN). Best working path selected: direct (#24) preferred, TURN-relayed (#25) only if all direct paths fail. #45 DTLS handshake on the selected path (keys derived from SDP fingerprints exchanged in signaling — relay cannot MITM).

6. **A2A tasks/send** — runner-A constructs A2A `tasks/send` envelope, includes #10 JWT in auth header. If #24 direct: over #33 WebRTC + #45 DTLS + #31 A2A. If #25 fallback: over #20 + #43 TURN as opaque #45 DTLS-encrypted ciphertext (relay sees ciphertext only).

7. **A2A receive** — runner-C's #16 A2A endpoint authenticates #10 JWT (signed by chepherd-daemon-Y which runner-C trusts), persists the task, generates knock marker.

8. **Knock injection** — runner-C writes knock to its locally-held #13 PTY master FD; kernel transfers via #29 (PTY #39 mechanism) to #14 slave; agent-C reads from stdin.

9. **Structured fetch** — agent-C sees knock, calls `chepherd.get_task(t-xxx)` via #21 / #30 MCP; runner-C's #17 MCP endpoint returns the full A2A task envelope (body, metadata, attachments). Agent-C processes.

10. **Async audit** — runner-A sends audit event to chepherd-daemon-X via #22; runner-C sends audit event to chepherd-daemon-Y via #22. Each daemon stores in its own #8 audit log.

### What did NOT happen in this flow

- ❌ No regex parsing of any stdout (no `@-relay`)
- ❌ No PTY FDs traversed pod boundaries (each runner holds its own master)
- ❌ No daemon-side PTY ownership (#1 holds no FDs in either org)
- ❌ No agent payload ever flowed through chepherd-daemon (#1 in either org)
- ❌ No agent payload ever flowed through relay (#2) in plain text — #45 DTLS end-to-end
- ❌ No org-X credential ever sent to runner-C — JWT was minted by chepherd-daemon-Y, signed by its own key
- ❌ No `chepherd-bridge` stdio adapter — agent talks HTTP-MCP to runner directly via #15

### Simpler cases (collapse to subsets of the above)

- **Same chepherd, same cluster, two pods** (agent-A and agent-B both managed by chepherd-daemon-X): skip the federation hop in step 2 (chepherd-daemon-X handles both sides locally, mints JWT itself); steps 3-4 may skip relay if pods can reach each other directly on cluster network. Steps 5-9 identical.
- **Same chepherd, same pod** (artificial — two agents in one pod): #24 collapses to a local loopback A2A call within one runner. Same envelope shape, same audit, same knock-and-fetch pattern. Useful only for testing.

## How to extend this doc

When new architecture is proposed, add components to the table above with the next available #. Each new component must have:
- Distinct name
- One of the 7 categories (or a new category if truly needed)
- One-line role
- Logical + physical location
- Cross-references to other # components

When new flows are designed, draft a sequence diagram in the same Mermaid + ASCII format, referencing the # of every component touched.

The discipline: **no architectural discussion uses an unnamed component.** If a name isn't in the table, it doesn't exist yet.
